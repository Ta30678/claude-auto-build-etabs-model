> **用途**：Boss 向潛在客戶/設計院展示簡報
> **時長**：20–30 分鐘
> **語言**：中文為主
> **頁數**：16 頁
> **設計風格**：深色背景 + 亮色重點（科技感），每頁文字 ≤ 5 行，多用圖示
> **故事線**：概念先行（70%）→ 應用案例（30%）

## 第 1 頁：封面

**標題**：用 AI Agent 打造智能工作流
**副標題**：從概念到落地——分層架構的力量

### 建議素材
- 公司 Logo（左上角或置中）
- 抽象科技背景（分層結構 / 金字塔意象，半透明處理）
- 日期、講者姓名

### 講者備註
> 簡短自我介紹（30 秒以內）。開場白：
> 「今天我不只要展示一個工具，而是一套方法論——如何用 AI Agent 架構，把任何需要專業判斷的工作流程，變成可靠、可重現的自動化系統。」

## 第 2 頁：AI Agent 是什麼？

**標題**：從 ChatBot 到 Agent——AI 的進化

### 要點

| | 💬 傳統 ChatBot | 🤖 AI Agent |
|---|----------------|-------------|
| 互動模式 | 被動問答——你問一句、它答一句 | 主動執行——給任務，自己拆解步驟完成 |
| 工具能力 | 只能產生文字 | 能讀寫檔案、執行程式、操作外部工具 |
| 協作方式 | 單一對話 | 多個 Agent 分工合作，各司其職 |
| 結果 | 一段回答 | 實際成果——檔案、模型、報告 |

### 建議素材
- 左右對比排版：左邊 ChatBot（對話泡泡），右邊 Agent（流程圖＋工具圖標）
- 中間用大箭頭標示「進化」

### 講者備註
> 「大家都用過 ChatGPT——輸入問題、拿到回答，但那只是 AI 的起點。AI Agent 是下一個階段：它不只回答問題，它能**動手做事**。讀檔案、寫程式、操作軟體、跟其他 Agent 協作——真正把工作完成。」

## 第 3 頁：Claude Code — AI 開發環境

**標題**：Claude Code：讓 Agent 落地的平台

### 要點
- **命令列工具（CLI）**——在終端機直接與 AI 對話、下達任務
- 核心能力：
  - 📂 讀寫任何檔案
  - 💻 執行程式碼
  - 🔧 操作外部工具與軟體
  - 🧩 可自訂**指令、Agent、技能**——完全根據你的需求量身打造
- 不是通用聊天工具，是**專業工作平台**

### 建議素材
- Claude Code 終端畫面截圖（展示 AI 執行任務的過程）
- 四個核心能力用圖標排列
- 底部對比：「通用 AI 聊天室 vs 專業 AI 工作站」

### 講者備註
> 「Claude Code 是我們整個系統的基礎——它讓 AI 不再只是聊天窗，而是一個真正的工作環境。你可以自訂指令告訴它你的規則、建立專屬 Agent 處理特定任務、定義技能讓知識可以跨專案重複使用。接下來我會逐層拆解這個架構。」

### 講者備註

## 第 5 頁：第 1 層——專案說明書

**標題**：第 1 層：AI 的工作手冊

### 要點
- 一份 **Markdown 文件**，定義 AI 在此專案中的所有規則
- 內容涵蓋：
  - 📖 專業術語定義（AI 要說你的語言）
  - 📏 工程規則與數值（精確到小數點）
  - 🔌 軟體 API 用法（哪些能用、怎麼用）
  - 🚫 禁止事項（哪些事絕不能做）
- 類比：就像新進員工的 **SOP 手冊**——還沒上工就先讀手冊

### 建議素材
- 左側：一份 Markdown 文件的截圖（模糊化處理，展示結構感）
- 右側：對應到的概念（規則、術語、API、禁止事項），用圖標表示
- 底部類比：📋 SOP 手冊 → AI 讀完就知道怎麼做

### 講者備註
> 「AI 再聰明，也需要知道你的規則是什麼。這份說明書就是 AI 的 SOP 手冊——它會讀這份文件，了解這個專案的所有術語、規則、能用的工具、不能做的事。就像新人第一天報到要先讀手冊，AI 每次啟動也先讀這份文件。」

---

## 第 6 頁：第 2 層——工作流命令

**標題**：第 2 層：一鍵啟動完整流程

### 要點
- 一個命令 = 一條完整的生產線
- 命令定義了：
  - 🔢 **Phase 順序**——先做什麼、後做什麼
  - ✅ **參數檢查**——輸入不完整就不開工
  - 👥 **團隊組成**——自動召集需要的 Agent
- 使用者只需一行指令，系統自動編排所有步驟
- 類比：工廠的**生產線按鈕**——按一下，整條產線啟動

### 建議素材
- 一個大按鈕圖標 → 觸發後展開為一條流程線（3-4 個 Phase）
- 每個 Phase 標明做什麼、由誰負責
- 底部類比：🏭 按鈕 → 生產線自動運轉

### 講者備註
> 「有了手冊，接下來要定義流程。一個工作流命令就像工廠的啟動按鈕——你按下去，它會按照預設的順序，先檢查輸入夠不夠、然後依序召集 Agent 開始工作。使用者不需要記住步驟，一行指令搞定。」

---

## 第 7 頁：第 3 層——AI Agent 團隊

**標題**：第 3 層：多專家分工

### 要點
- 每個 Agent 是一位**專家**，有明確的角色定義
- 關鍵設計：
  - 🎭 **角色定義**——每位專家只負責自己擅長的事
  - 🔧 **工具權限**——每位專家有自己能使用的工具
  - 💬 **直接溝通**——專家之間可以互傳訊息、交叉確認
- 類比：像一個**專案小組**——讀圖專家、定位專家、整合專家，各司其職

### 建議素材
- 三張角色卡片排列，中間用雙向箭頭連接（表示溝通）
- 每張卡片標：角色名稱（概念描述）、負責事項、可用工具
- 底部類比：👥 專案小組 → 分工合作

### 講者備註
> 「一個人做所有事，不如三個專家分工合作。每個 Agent 有自己的角色定義——只做自己擅長的部分。他們有各自的工具權限，也能互相溝通確認。就像你組一個專案小組，每個人有明確的分工，但遇到問題可以互相討論。」

---

## 第 8 頁：第 4 層——專業技能模組

**標題**：第 4 層：可複用的知識庫

### 要點
- **技能模組** = 封裝好的專業知識（Domain Knowledge）
- 特色：
  - 📚 **跨專案共享**——同一份專業知識可用於不同案件
  - 🔄 **隨時更新**——規則變了改一處，全案生效
  - 📋 **標準化**——所有 Agent 引用同一套知識，不會各說各話
- 類比：像專業**教科書/參考手冊**——每個工程師桌上都有一本，翻開就能查

### 建議素材
- 中央一本「技能手冊」圖標，四周放射出連到不同專案的箭頭
- 或：書架圖——不同技能模組排在書架上，Agent 取用
- 底部類比：📖 專業手冊 → 知識統一、隨查隨用

### 講者備註
> 「技能模組是讓系統越用越強的關鍵。把專業知識寫成標準化的模組——比方說『如何讀結構圖』、『斷面命名規則』——這些知識寫一次，所有專案、所有 Agent 都能共用。規則更新了？改一個地方，全部生效。」

---

## 第 9 頁：第 5 層——確定性執行腳本

**標題**：第 5 層：AI 理解，程式執行

### 要點
- **AI 負責「讀懂需求」→ 程式負責「精準動作」**
- 確定性 = **同樣的輸入，永遠得到同樣的結果**
- 所有工程規則**寫在程式碼裡**，不由 AI 即興決定
- 類比：**AI 是建築師，腳本是施工隊**——建築師畫好圖，施工隊照圖施工，不會自己改設計

### 建議素材
- 左側：AI 圖標（腦）→ 產出一份規格書
- 右側：程式碼圖標（齒輪）→ 讀規格書 → 產出成果
- 中間連接：規格書（config / 設定檔）
- 底部類比：🏗️ 建築師畫圖 → 施工隊照圖蓋

### 講者備註
> 「這是我們架構中最關鍵的設計理念——把 AI 的『理解力』跟程式的『執行力』分開。AI 很擅長讀懂圖紙、理解需求，但你不會想讓它即興決定工程參數。所以我們讓 AI 把理解的結果寫成規格書，再由確定性的程式碼照規格執行。同一份規格，跑一百次，結果一模一樣。」

---

## 第 10 頁：第 6 層——自動化驗證

**標題**：第 6 層：每次執行，自動品質檢查

### 要點
- 每次執行完成後，**自動跑品質檢測**
- 驗證項目涵蓋：
  - ✅ 數值正確性（參數、設定值）
  - ✅ 完整性（該有的東西都有）
  - ✅ 一致性（與規格書吻合）
- 不通過 → **精確指出哪裡有問題**，不是模糊的「有錯」
- 類比：**出廠品管**——每一件產品離開產線前都要過 QC

### 建議素材
- 一個檢測報告畫面（綠色勾勾列表）
- 或：生產線末端的 QC 站圖示，檢查項目逐一打勾
- 底部類比：🏭 QC 品管站 → 不合格不出貨

### 講者備註
> 「人會漏檢，程式不會。每次執行完，系統自動驗證所有關鍵項目——數值對不對、東西齊不齊、跟規格一不一致。任何一項不通過，馬上告訴你是哪裡出問題，不需要人工逐項檢查。這就是為什麼我們敢說品質穩定。」

## 第 11 頁：設計哲學

**標題**：各取所長——AI 理解 + 程式執行

### 要點

```
┌──────────────────────────────────────────────┐
│                                              │
│   AI 擅長              程式擅長              │
│   ─────────            ─────────             │
│   理解自然語言          精準執行指令            │
│   判讀圖紙              數值計算零誤差          │
│   處理模糊資訊          同輸入 = 同輸出         │
│   跨領域知識整合         毫秒級大量重複操作       │
│                                              │
│         ┌──────────────────┐                 │
│         │  規格書 / 設定檔   │                 │
│         │   （結構化資料）    │                 │
│         └──────────────────┘                 │
│          AI 產出 ←→ 程式讀取                   │
│                                              │
└──────────────────────────────────────────────┘
```

- **核心原則**：讓 AI 做 AI 擅長的事，讓程式做程式擅長的事
- 兩者的介面 = **結構化的設定檔（規格書）**
- 結果：既有 AI 的靈活理解力，又有程式的精準執行力

### 建議素材
- 左右對比：AI（腦圖標）vs 程式（齒輪圖標），中間一份文件連接
- 或：維恩圖——兩個圓重疊處寫「設定檔」
- 設計要精緻——這頁是概念篇的總結

### 講者備註
> 「這就是整套系統的設計哲學——不是讓 AI 取代人，也不是讓 AI 取代程式，而是讓對的角色做對的事。AI 負責理解，這是它最強的地方；程式負責執行，保證精準不出錯。兩者之間用一份標準化的設定檔溝通。這個原則適用於任何領域——接下來我會用一個實際案例來說明。」

## 第 12 頁：應用案例——結構建模

**標題**：案例：AI 驅動的 ETABS 自動建模

### 要點
- **痛點**：一棟 20 層建築的 ETABS 建模，手動需要 **2–3 天**
  - 讀圖 → Grid → 斷面 → 柱梁牆 → 配筋 → 載重 → 檢查，全程手動
- **解決方案**：套用六層架構 → 全程 **15–20 分鐘**，零手動
- 三階段流程：

```
┌─────────────┐     ┌─────────────┐     ┌─────────────────────┐
│  AI 讀圖     │     │  AI 整合     │     │   程式自動建模        │
│  判讀結構圖   │ ──→ │  生成規格書   │ ──→ │   11 步確定性執行     │
│  ~3 分鐘     │     │  ~2 分鐘     │     │   ~10 分鐘           │
└─────────────┘     └─────────────┘     └─────────────────────┘
   AI 理解圖紙         AI 整合輸出          程式精準建模
```

### 建議素材
- 上半：痛點區（工程師加班圖 + 時間軸標「2–3 天手動」）
- 下半：解決方案區（三段式流程圖 + 「15–20 分鐘」大字）
- 對比效果要強烈

### 講者備註
> 「我們剛才講的六層架構不是紙上談兵——這是我們實際落地的案例。結構建模是一個典型的『需要專業判斷 + 大量重複操作』的工作。AI 讀圖、理解圖紙內容、生成規格書，然後確定性的程式碼按照規格把模型建出來。從 2–3 天變成 15–20 分鐘。」

## 第 14 頁：效率對比

## 第 15 頁：更多應用可能性

**標題**：不只是結構建模——架構可移植到任何場景

### 要點
- 六層架構的核心邏輯：**AI 理解 + 確定性執行 + 自動驗證**
- 只要符合這個模式的工作流，都能套用：

| 應用場景 | AI 理解 | 確定性執行 | 自動驗證 |
|----------|---------|-----------|----------|

## 附錄：PPT 設計風格指引

- **色調**：深色（深藍/深灰）背景 + 亮色（白/橘/藍綠）重點文字
- **字體**：標題大字（≥ 36pt）、內文簡潔（≥ 20pt）
- **原則**：每頁文字 ≤ 5 行，多用圖示、表格、流程圖
- **流程圖**：用大箭頭，一目瞭然；避免複雜連線
- **數字**：重要數據用超大字體突出（如「15 分鐘」「0 錯誤」）
- **動畫**：建議只用簡單的「出現」效果，不要花俏轉場
- **概念頁（1–11）**：著重圖示和類比，避免技術細節
- **應用頁（12–16）**：著重數據和對比，用具體數字說話
